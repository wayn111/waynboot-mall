# 商品订单第二阶段优化说明

## 背景

本轮围绕后续优化方向 `1 2 3 4 5` 做收口：订单状态机、商品高并发读缓存、MQ 消费可靠性、事务边界收缩和测试覆盖。目标是在不恢复批量下单复杂度的前提下，提升核心链路的可读性、并发安全性和后续扩展能力。

## 范围

- 订单链路：集中维护订单状态流转规则，支付、取消、发货、退款、确认收货统一复用状态机校验。
- 商品链路：移动端商品详情增加 Redis 聚合缓存，商品写入和库存变更后主动失效缓存。
- MQ 消费：单条消费模板区分业务失败和 ACK 后幂等标记失败，避免对已 ACK 消息继续 NACK。
- 事务边界：支付成功后的虚拟销量更新改为异步后置动作，不阻塞支付回调主链路。
- 测试覆盖：补充状态机、商品详情缓存、支付后置动作和 MQ ACK 后异常分支测试。

## 核心逻辑调整

### 订单状态机

新增 `OrderStateTransitionSupport`，集中维护合法流转：

- `STATUS_CREATE` 可以进入 `STATUS_PAY`、`STATUS_CANCEL`、`STATUS_AUTO_CANCEL`。
- `STATUS_PAY` 可以进入 `STATUS_REFUND`、`STATUS_SHIP`。
- `STATUS_REFUND` 可以进入 `STATUS_REFUND_CONFIRM`。
- `STATUS_SHIP` 可以进入 `STATUS_CONFIRM`、`STATUS_AUTO_CONFIRM`。

支付回调、用户取消、用户退款申请、管理端发货、管理端退款确认和确认收货都通过该组件校验或判断状态流转。数据库更新仍保留当前状态条件，防止并发请求覆盖状态。

### 商品详情缓存

`GoodsDetailServiceImpl` 新增 Redis 缓存读取和回填，缓存 Key 为 `GOODS_DETAIL_CACHE`。商品详情包含商品信息、规格、SKU 库存和属性，是移动端高频读接口，缓存可以降低并发读压力。

由于详情包含库存，以下写路径会主动删除对应商品详情缓存：

- 商品保存、更新、删除。
- 虚拟销量更新。
- 下单库存扣减。
- 取消、退款等库存回补。

### MQ 消费可靠性

`AbstractSingleMessageConsumer` 继续保持模板方法模式，统一解析消息、重复消费判断、业务回调、ACK/NACK 和幂等标记。

本轮新增 `MessageConsumedMarkException`，用于表达“业务已成功且 broker 已 ACK，但 Redis 幂等标记写入失败”。该场景不能再 NACK 同一 deliveryTag，否则会对已确认消息执行无效且误导性的失败处理；当前策略是记录错误日志，后续可接入告警和补偿任务。

### 支付后置动作异步化

`PaymentPostActionSupport.handleOrderPaid` 不再在支付回调线程内同步查询订单商品和更新虚拟销量，而是投递到 `commonThreadPoolTaskExecutor` 异步执行。支付状态更新仍在主链路内完成，虚拟销量属于可补偿的派生数据，失败只记录日志。

## 并发与一致性变化

- 订单状态变更统一由状态机定义合法来源，数据库更新继续使用当前状态条件，避免并发支付、取消、退款、发货之间互相覆盖。
- 商品详情缓存采用写后失效而不是写后更新，避免复杂聚合数据在事务内重复构建。
- 库存扣减和回补成功后删除详情缓存，保证后续详情请求重新读取最新库存。
- MQ 消费明确区分业务失败和 ACK 后标记失败，避免错误地对已 ACK 消息执行 NACK。
- 支付回调主链路缩短，只保留状态更新和必要校验，派生数据异步处理。

## 验证方式

已执行：

```bash
mvn "-Dmaven.repo.local=.m2/repository" test
```

验证结果：

- Reactor 全模块 `BUILD SUCCESS`。
- 覆盖订单状态机、取消补偿、支付回调幂等、商品详情缓存、支付后置异步和 MQ 消费模板。

## 非目标

- 不恢复批量下单接口。
- 不引入新的库存预热或 Redis 库存扣减模型。
- 不改变支付渠道验签协议。
- 不改变对外接口路径和返回结构。

## 后续计划

- 对订单状态机增加操作审计表，记录状态来源、目标状态、操作者和失败原因。
- 对商品详情缓存加入热点商品预热和缓存击穿保护。
- 对 MQ ACK 后幂等标记失败接入告警，并评估是否增加消费成功流水表。
- 对支付后置动作拆出领域事件，统一承载积分、通知、销量、ES 同步等派生动作。
- 增加压测脚本，分别覆盖商品详情接口、下单接口和支付回调接口。

## 方法论补充

后续交易域优化统一参考 `ECOMMERCE_TRADE_SYSTEM_METHODOLOGY.md`，避免只围绕单点功能继续堆代码。订单、库存、支付、MQ、分账、对账、分表和高可用要作为一条完整交易链路评估：

- 同步链路只保留下单核心校验、订单核心落库、库存冻结和必要幂等控制。
- 派生副作用优先走本地消息表、MQ 和补偿任务，避免支付回调或下单接口被额外副作用拖慢。
- 库存一致性以 MySQL 条件更新兜底；Redis 预扣只能作为削峰能力，不能替代数据库最终一致性。
- 状态并发以订单状态机和数据库当前状态条件更新共同控制，支付成功、超时取消、退款确认等动作不能绕过状态机。
- 数据规模治理要提前考虑订单分表、支付流水、库存流水和对账差异表，避免等单表膨胀后再被动拆分。
